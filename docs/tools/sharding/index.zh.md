# 分片

**分片**是一种数据库架构模式，它将大型、逻辑统一的数据库集水平划分为更小、独立的数据库，称为*分片*。每个分片托管在单独的服务器实例上，采用“无共享”架构运行。分片通过将数据和工作负载分布到多个节点，突破了单台机器的垂直扩展极限。

## 什么是分片

分片基于确定性的*分片键*将数据划分为多个块。每个分片包含数据的一个子集（例如，某个`user_id`范围的所有行），并负责处理其分区的读写操作。通过路由层（应用逻辑、代理或数据库路由器），整个系统对客户端呈现为一个逻辑数据库。

## 为什么使用分片

| 优势 | 描述 |
|------|------|
| **水平可扩展性** | 写入和读取吞吐量随着分片的增加而线性提升。 |
| **高可用性与故障隔离** | 单个分片故障仅影响部分用户，其他分片可继续服务。 |
| **并行性** | 涉及多个分片的查询可并行处理，降低延迟。 |
| **地理分布** | 数据可放置到特定用户群体附近，减少网络往返。 |
| **运维隔离** | 维护、备份和模式变更可逐个分片进行。 |

当单个数据库无法再承受负载时——通常是在垂直扩展（更强的CPU、更大的内存）变得成本过高或遇到硬件限制之后——分片就变得至关重要。

## 分片架构

分片可以在多个层面实现：

### 1. 应用层分片（手动）

应用包含路由逻辑（例如`hash(user_id) % num_shards`）。每个分片是一个标准数据库，无需额外软件。  
**优点：** 初始简单，无需中间件。  
**缺点：** 脆弱；重新分片需要修改代码；跨分片查询极其困难。  
**现状：** 如今被认为是新项目的反模式。

### 2. 中间件/代理层分片（例如 Vitess、Citus）

透明代理拦截SQL查询并将其路由到正确的分片。

- **Vitess**（用于MySQL）：部署`vtgate`（代理）和每个分片的`vttablet`，由etcd/zookeeper拓扑管理。
- **Citus**（用于PostgreSQL）：一个扩展，将PostgreSQL集群转变为分布式数据库。

**优点：** SQL透明、自动重新分片（Vitess）、跨分片连接（Citus）。  
**缺点：** 增加了一层复杂性；某些查询可能无法执行或变慢。

### 3. 数据库原生分片（例如 MongoDB、Cassandra、Druid）

数据库引擎内部处理数据分布。开发者提供分片键，系统管理数据放置和路由。

- **MongoDB**：分片集群，包含`mongos`路由器和配置服务器。
- **Cassandra**：通过主键定义中的分区键进行分区；一致性哈希自动分布行。

**优点：** 无需外部代理；支持自动平衡等功能。  
**缺点：** 必须仔细设计围绕分片键的数据模型；跨分片操作受限或缺失。

### 4. 云托管分片（例如 Amazon DynamoDB、Azure Cosmos DB、Google Cloud Spanner）

提供商完全抽象分片管理。你在创建表时选择分区键，云平台自动分裂、迁移和平衡数据。

**优点：** 零运维开销；自动扩展。  
**缺点：** 供应商锁定；对于大量工作负载成本可能较高；无法直接控制分片放置。

## 安装与基本使用

以下是两种最常见分片实现的具体示例。

### MongoDB分片

**安装/设置**

- 部署**配置服务器副本集**（CSRS），存储集群元数据。
- 部署**分片副本集**（每个分片至少是一个节点，但通常使用副本集以实现高可用）。
- 部署一个或多个**`mongos`路由器**，处理应用查询。

以下命令（在`mongos`上运行）启用分片并对集合进行分片：

```javascript
// 对数据库启用分片
sh.enableSharding("ecommerce");

// 使用哈希分片键（推荐用于均匀分布）对集合进行分片
sh.shardCollection(
  "ecommerce.orders",
  { "order_id": "hashed" }
);
```

使用哈希分片键后，文档均匀分布在各个分片上。包含分片键的查询会直接路由到正确的分片：

```javascript
// 高效查询 – 仅访问单个分片
db.orders.find({ "order_id": UUID("123e4567-e89b-12d3-a456-426614174000") })
```

跨分片查询（例如不包含分片键的聚合）会面向所有分片，可能导致性能下降。

### Citus（PostgreSQL扩展）

**安装**

1. 在协调器节点和所有工作节点上安装`citus`扩展。
2. 将工作节点添加到协调器：
   ```sql
   SELECT citus_add_node('worker-node-1', 5432);
   SELECT citus_add_node('worker-node-2', 5432);
   ```

**基本使用**

通过指定分布列（分片键）来分布表：

```sql
-- 在协调器上创建表
CREATE TABLE orders (
    order_id    BIGSERIAL,
    user_id     INT,
    product_id  INT,
    quantity    INT,
    PRIMARY KEY (order_id, user_id)
);

-- 基于user_id在工作节点间分布表
SELECT create_distributed_table('orders', 'user_id');
```

Citus重写SQL以命中相关分片。对`user_id`进行过滤的查询会发送到单个工作节点：

```sql
-- 单分片查询
SELECT * FROM orders WHERE user_id = 42;
```

对于两个在相同分布键上共置的表之间的连接，Citus可以高效执行：

```sql
-- 共置示例：orders和order_items按user_id分布
SELECT create_distributed_table('orders', 'user_id');
SELECT create_distributed_table('order_items', 'user_id');
-- JOIN现在在每个分片上本地执行
SELECT o.order_id, oi.product_id
FROM orders o JOIN order_items oi USING (order_id)
WHERE o.user_id = 42;
```

## 关键设计决策

### 1. 选择分片键

分片键是最关键的决策。它必须：

- **均匀分布数据**以避免热点。
- **匹配查询模式**，使常见查询能够路由到单个分片。
- **具有高基数**（大量不同值），以允许均匀分割。

**错误选择：** 单调递增的值（例如时间戳、自增ID）会导致所有新写入都进入最后一个分片。  
**较佳选择：** 用户ID、哈希列或结合高基数和常用过滤列的复合键。

### 2. 跨分片操作

跨多个分片的JOIN、事务和聚合要么代价高昂，要么不被支持。缓解策略包括：

- **反规范化**，将相关数据保存在同一分片。
- **共置**（Citus）或**文档嵌入**（MongoDB），在同一个分片中存储层次化数据。
- **应用端协调**用于跨分片事务（一般不推荐）。

### 3. 重新分片

添加或移除分片需要重新分布数据。现代系统提供内置机制：

- **MongoDB Balancer** 自动在分片间移动数据块。
- **Vitess Reshard** 使用`MoveTables`工作流分割分片。
- **云服务** 透明地处理分裂。

手动重新分片（在应用层分片中）极其困难且容易出错。

## 现代状态

行业正在远离手动分片。**NewSQL**数据库（CockroachDB、YugabyteDB、Google Spanner）在标准SQL接口背后完全抽象了分片管理，提供跨分片的ACID事务和连接。大多数云数据库（DynamoDB、Cosmos DB）提供无服务器分片。然而，分片的核心概念仍然是所有水平可扩展分布式数据库的基础。

对于新项目，优先选择这些方案而不是构建自己的分片层。如果你需要SQL和强一致性，可以考虑Citus或Spanner；如果面向文档的灵活性和海量吞吐量是首要需求，MongoDB或DynamoDB是不错的选择。

## 总结

分片是实现Web规模性能的强大工具，但也引入了复杂性。通过了解架构选项、选择良好的分片键并利用现代管理工具，你可以扩展数据层，而无需重新发明轮子。
