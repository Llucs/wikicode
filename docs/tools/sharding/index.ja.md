# Sharding

**Sharding** は、大規模で論理的に統一されたデータセットを水平分割して、*シャード*と呼ばれる小さな独立したデータベースにするデータベースアーキテクチャパターンです。各シャードは、"shared-nothing" アーキテクチャで動作する個別のサーバーインスタンスにホストされます。Sharding は、データとワークロードを多くのノードに分散することで、単一マシンの垂直スケーリングの限界を打破します。

## シャーディングとは

シャーディングは、決定論的な *シャードキー* に基づいてデータをチャンクに分割します。各シャードはデータのサブセット（例：特定の `user_id` 範囲のすべての行）を含み、そのパーティションの読み取りと書き込みを担当します。システム全体は、ルーティングレイヤー（アプリケーションロジック、プロキシ、データベースルーター）を通じてクライアントに1つの論理データベースとして見えます。

## なぜシャーディングするのか？

| 利点 | 説明 |
|---------|-------------|
| **水平スケーラビリティ** | シャードを追加するにつれて、書き込みと読み取りのスループットが線形に拡張されます。 |
| **高可用性と障害分離** | 単一のシャード障害は一部のユーザーにのみ影響し、他のシャードはサービスを継続します。 |
| **並列性** | 複数のシャードにまたがるクエリは並列化でき、レイテンシが改善されます。 |
| **地理的分散** | データを特定のユーザー集団の近くに配置し、ネットワークのラウンドトリップを削減できます。 |
| **運用の分離** | メンテナンス、バックアップ、スキーマ変更をシャードごとに順次実行できます。 |

シャーディングは、単一のデータベースが負荷を処理できなくなった場合に不可欠です。多くの場合、垂直スケーリング（より大きなCPU、より多くのRAM）がコスト的に非効率になるか、ハードウェアの限界に達した後です。

## シャーディングアーキテクチャ

シャーディングはいくつかのレイヤーで実装できます。

### 1. アプリケーションレベル（手動）

アプリケーションにルーティングロジック（例：`hash(user_id) % num_shards`）が含まれます。各シャードは追加ソフトウェアなしの標準的なデータベースです。  
**長所:** シンプルに始められ、ミドルウェアが不要。  
**短所:** 脆い；リシャーディングにはコード変更が必要；クロスシャードクエリが非常に困難。  
**状況:** 今日では新規プロジェクトにはアンチパターンと見なされています。

### 2. ミドルウェア／プロキシレベル（例：Vitess、Citus）

透過的なプロキシがSQLクエリをインターセプトし、適切なシャードにルーティングします。

- **Vitess** for MySQL：`vtgate`（プロキシ）+ `vttablet`をシャードごとにデプロイし、etcd/zkトポロジーで管理。
- **Citus** for PostgreSQL：Postgresクラスターを分散データベースにする拡張機能。

**長所:** SQL透過性、自動リシャーディング（Vitess）、シャード間JOIN（Citus）。  
**短所:** 追加の複雑性；一部のクエリが不可能または遅くなる。

### 3. データベースネイティブ（例：MongoDB、Cassandra、Druid）

データベースエンジンが内部的に分散を処理します。開発者はシャードキーを指定し、システムがデータ配置とルーティングを管理します。

- **MongoDB**：`mongos`ルーターと構成サーバーを持つシャーディングクラスター。
- **Cassandra**：プライマリキーのパーティションキーによるパーティショニング；コンシステントハッシュが行を自動分散。

**長所:** 外部プロキシ不要；自動バランシングなどの機能。  
**短所:** シャードキーに基づくデータモデルの慎重な設計が必要；クロスシャード操作は制限または非対応。

### 4. クラウドマネージド（例：Amazon DynamoDB、Azure Cosmos DB、Google Cloud Spanner）

プロバイダーがシャード管理を完全に抽象化します。テーブル作成時にパーティションキーを選択し、クラウドプラットフォームが自動的にデータの分割、移行、バランシングを行います。

**長所:** 運用オーバーヘッドゼロ；自動スケール。  
**短所:** ベンダーロックイン；大規模ワークロードではコストが高くなる可能性；シャード配置の直接制御不可。

## インストールと基本使用法

以下は、最も一般的なシャーディング実装のうち2つの具体的な例です。

### MongoDB シャーディング

**インストール／セットアップ**

- クラスターメタデータを格納する **config server replica set**（CSRS）をデプロイ。
- **シャードレプリカセット**をデプロイ（各シャードは少なくとも1ノードだが、通常は高可用性のためにレプリカセット）。
- アプリケーションクエリを処理する1つ以上の **`mongos` ルーター**をデプロイ。

次のコマンド（`mongos`に対して実行）は、シャーディングを有効にし、コレクションをシャーディングします。

```javascript
// データベースでシャーディングを有効にする
sh.enableSharding("ecommerce");

// ハッシュ化されたシャードキーを使用してコレクションをシャーディング（均等分散に推奨）
sh.shardCollection(
  "ecommerce.orders",
  { "order_id": "hashed" }
);
```

ハッシュ化されたシャードキーを使用すると、ドキュメントはシャード間で均等に分散されます。シャードキーを含むクエリは、正しいシャードに直接ルーティングされます。

```javascript
// 効率的なクエリ – 単一シャードに送られる
db.orders.find({ "order_id": UUID("123e4567-e89b-12d3-a456-426614174000") })
```

クロスシャードクエリ（例：シャードキーなしの集計）はすべてのシャードに分散され、パフォーマンスを低下させる可能性があります。

### Citus (PostgreSQL 拡張)

**インストール**

1. コーディネーターノードとすべてのワーカーノードに `citus` 拡張機能をインストール。
2. ワーカーノードをコーディネーターに追加：
   ```sql
   SELECT citus_add_node('worker-node-1', 5432);
   SELECT citus_add_node('worker-node-2', 5432);
   ```

**基本使用法**

配布カラム（シャードキー）を指定してテーブルを分散：

```sql
-- コーディネーターにテーブルを作成
CREATE TABLE orders (
    order_id    BIGSERIAL,
    user_id     INT,
    product_id  INT,
    quantity    INT,
    PRIMARY KEY (order_id, user_id)
);

-- user_id に基づいてワーカー間でテーブルを分散
SELECT create_distributed_table('orders', 'user_id');
```

Citus はSQLを書き換えて関連シャードにアクセスします。`user_id` でフィルタリングするクエリは単一ワーカーに送られます。

```sql
-- 単一シャードクエリ
SELECT * FROM orders WHERE user_id = 42;
```

同じ配布キーでコロケーションされた2つのテーブル間のJOINは、Citus で効率的に実行できます。

```sql
-- コロケーション例: orders と order_items を user_id で分散
SELECT create_distributed_table('orders', 'user_id');
SELECT create_distributed_table('order_items', 'user_id');
-- JOIN は各シャードでローカルに実行される
SELECT o.order_id, oi.product_id
FROM orders o JOIN order_items oi USING (order_id)
WHERE o.user_id = 42;
```

## 主要な設計上の決定

### 1. シャードキーの選択

シャードキーは最も重要な決定事項です。以下の条件を満たす必要があります。

- **データを均等に分散**してホットスポットを回避する。
- **クエリパターンに一致**し、一般的なクエリを単一シャードにルーティングできる。
- **高いカーディナリティ**（多くの異なる値）を持ち、均等な分割を可能にする。

**悪い選択:** 単調増加する値（例：タイムスタンプ、オートインクリメントID）は、すべての新規書き込みが最後のシャードに集中します。  
**良い選択:** ユーザーID、ハッシュ化されたカラム、または高いカーディナリティと頻繁にフィルタリングされるカラムを組み合わせた複合キー。

### 2. クロスシャード操作

複数のシャードにまたがるJOIN、トランザクション、集計は非常に高コストかサポートされていません。軽減策：

- **非正規化**により関連データを同じシャードに保持。
- **コロケーション**（Citus）や**ドキュメント埋め込み**（MongoDB）で階層データを同じシャードに格納。
- **アプリケーション側での調整**によるマルチシャードトランザクション（推奨は稀）。

### 3. リシャーディング

シャードの追加または削除にはデータの再分散が必要です。現代のシステムは組み込みのメカニズムを提供します。

- **MongoDB バランサー**は自動的にシャード間でチャンクを移動。
- **Vitess Reshard** は `MoveTables` ワークフローを使用してシャードを分割。
- **クラウドサービス**は透過的に分割を処理。

手動リシャーディング（アプリケーションレベルのシャーディング）は非常に困難でエラーが発生しやすいです。

## 現代の状況

業界は手動シャーディングから離れつつあります。**NewSQL** データベース（CockroachDB、YugabyteDB、Google Spanner）は、標準SQLインターフェースの背後でシャード管理を完全に抽象化し、シャード間でのACIDトランザクションとJOINを提供します。ほとんどのクラウドデータベース（DynamoDB、Cosmos DB）はサーバーレスシャーディングを提供しています。しかしながら、シャーディングの核となる概念は、すべての水平スケーラブルな分散データベースの基盤であり続けています。

新規プロジェクトでは、独自のシャーディングレイヤーを構築するよりも、これらのいずれかを選択することをお勧めします。SQLと強い一貫性が必要な場合は、Citus または Spanner を検討してください。ドキュメント指向の柔軟性と大規模なスループットが最優先の場合は、MongoDB または DynamoDB が優れた選択肢です。

## まとめ

シャーディングはウェブスケールのパフォーマンスを実現するための強力なツールですが、複雑さをもたらします。アーキテクチャのオプションを理解し、適切なシャードキーを選択し、最新の管理ツールを活用することで、車輪の再発明をすることなくデータレイヤーをスケールできます。
