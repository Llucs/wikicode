# Immutable Infrastructure: 完全ガイド

## Immutable Infrastructure とは？

Immutable Infrastructure は、サーバーやコンテナがプロビジョニングされた後には決して変更されないデプロイメントモデルです。更新（パッチ、構成変更、コードリリース）が必要になった場合は、実行中のインスタンスを破棄し、標準化されたバージョン管理されたアーティファクト（「ゴールデンイメージ」やコンテナイメージと呼ばれる）からまったく新しいインスタンスを作成します。

このアプローチは、**可変インフラストラクチャ（mutable infrastructure）** とは対照的です。可変インフラストラクチャでは、オペレーターが SSH でライブサーバーにログインしてパッチを適用したり、構成管理ツール（Ansible、Chef、Puppet など）を実行したりします。可変インフラストラクチャはしばしば「構成ドリフト」や「スノーフレークサーバー」、つまり徐々に個別的で再現不可能になってしまう環境を引き起こします。

Immutable Infrastructure はサーバーを「cattle, not pets（ペットではなく家畜）」として扱います。つまり、使い捨て可能で、番号付けされ、簡単に交換できる存在です。システムへのあらゆる変更は、その場での修正ではなく、完全な再デプロイをトリガーします。

## なぜ Immutable Infrastructure なのか？

Immutable Infrastructure を採用する主な動機は、構成ドリフトとそれによって環境間で生じるばらつきを排除することです。利点は次のとおりです。

- **再現性（Reproducibility）：** すべてのデプロイはまったく同じアーティファクトから開始されるため、開発、ステージング、本番環境が同一であることが保証されます。
- **簡潔さ（Simplicity）：** 以前のイメージバージョンを再デプロイするだけでロールバックが簡単に行えます。
- **セキュリティ（Security）：** 本番インスタンスへの SSH アクセスが不要になり、攻撃対象領域が減少します。どのイメージがいつ実行されたかという監査証跡が明確になります。
- **スケーリング（Scaling）：** Auto Scaling グループやオーケストレーター（例：Kubernetes）は、既知の正常なイメージから新しいインスタンスを起動できるため、すべてのノードが均一になります。
- **使い捨て可能性（Disposability）：** インスタンスを停止して置き換えても可用性に影響がなく、シームレスなブルー/グリーンデプロイやカナリアデプロイが可能になります。

## 主要な原則と特徴

| 原則 | 説明 |
|-----------|-------------|
| **再現性** | すべての環境は同じバージョン管理されたアーティファクトから生成されます。 |
| **使い捨て可能性** | インスタンスは家畜であり、必要に応じて破棄および再作成できます。 |
| **アトミックデプロイ** | 更新はスタック全体を入れ替えることで行われ、その場でのパッチ適用は行われません。 |
| **シンプルなロールバック** | 以前の状態に戻すには、古いアーティファクトを再デプロイするだけです。 |
| **冪等性** | 同じアーティファクトを複数回デプロイしても、同一の結果が得られます。 |
| **ライブパッチなし** | 構成管理はイメージのビルド時のみに適用され、実行時には適用されません。 |

## 歴史と起源

- **2012年 – 「Pets vs. Cattle（ペット vs 家畜）」：** このアナロジーは、CloudScaling の Randy Bias 氏と Microsoft の Bill Baker 氏によって広められました。ペットはユニークで手作業で世話が必要ですが、家畜は番号付けされ、標準化され、簡単に交換できます。
- **2013年 – Chad Fowler 氏のブログ記事「Immutable Infrastructure」** でこの用語が正式に定義されました。
- **2013年 – Docker の登場：** コンテナはイミュータビリティに最適な手段となりました。つまり、エフェメラルで、標準化され、イメージから構築されます。
- **2014年 – HashiCorp Packer：** 単一のテンプレートから複数のクラウドプロバイダー（AWS AMI、Azure VHD、VMware）向けに同一のマシンイメージを作成することが実用的になりました。
- **2015年～現在 – Kubernetes、Terraform、CI/CD パイプライン：** これらのツールにより、イミュータブルデプロイはクラウドネイティブアプリケーションの業界標準になりました。

## ツールエコシステム

Immutable Infrastructure は **パラダイム** であり、単一のソフトウェアパッケージではありません。次の表に主要なツールとそのインストール方法を示します。

| レイヤー | ツール | インストール | 目的 |
|-------|------|--------------|---------|
| **イメージビルダー** | HashiCorp Packer | `brew install packer` / バイナリをダウンロード | ゴールデン VM/AMI を作成 |
| **コンテナイメージ** | Docker / Podman | `brew install docker` / `apt install docker.io` | コンテナイメージをビルド |
| **イメージレジストリ** | Docker Hub / ECR / GCR | クラウドプロバイダーのコンソール/CLI セットアップ | イミュータブルアーティファクトを保存＆バージョン管理 |
| **IaC / オーケストレーション** | Terraform / Pulumi / Kubernetes | `brew install terraform` / `kubectl` | イミュータブルリソースをデプロイ |
| **CI/CD** | GitLab CI / GitHub Actions | ランナーを構成 | ビルドとデプロイを自動化 |
| **シークレット注入** | HashiCorp Vault / AWS Secrets Manager | Vault エージェントまたは CSI ドライバーをインストール | 起動時にシークレットを注入（ベイクしない） |

> **注記：** 従来の構成管理ツール（Ansible、Chef、Puppet）も役割を果たしますが、それは **イメージのビルド段階のみ** です。Packer のプロビジョナー内や Dockerfile 内で使用され、実行中の本番インスタンスに対して使用されることは決してありません。

## 基本的な使用例

ここでは、イミュータブルの原則を使用して AWS に Nginx Web サーバーをデプロイする典型的なワークフローを見ていきます。

### ステップ 1: Packer でゴールデンイメージをビルドする

Packer テンプレート（例：`web.pkr.hcl`）を作成します：

```hcl
# web.pkr.hcl
source "amazon-ebs" "web" {
  ami_name      = "nginx-web-{{timestamp}}"
  source_ami    = "ami-0c02fb55956c7d316"   # Ubuntu 22.04 LTS
  instance_type = "t2.micro"
  region        = "us-east-1"
  ssh_username  = "ubuntu"
}

build {
  sources = ["source.amazon-ebs.web"]

  provisioner "shell" {
    inline = [
      "sudo apt-get update -y",
      "sudo apt-get install nginx -y",
      "sudo systemctl enable nginx"
    ]
  }
}
```

イメージをビルドします：

```bash
packer build web.pkr.hcl
```

出力は一意の AMI ID（例：`ami-0abc123def456`）です。これがイミュータブルなアーティファクトになります。

### ステップ 2: イミュータブルイメージからインスタンスをデプロイする

Terraform を使用します（`main.tf`）：

```hcl
# main.tf
resource "aws_instance" "web" {
  ami           = "ami-0abc123def456"
  instance_type = "t2.micro"

  tags = {
    Name = "immutable-web-v1"
  }
}
```

構成を適用します：

```bash
terraform apply
```

ゴールデン AMI から EC2 インスタンスが 1 つ起動されます。クラッシュした場合でも、同じイメージから新しいインスタンスが起動されるため、構成ドリフトは発生しません。

### ステップ 3: 新しいバージョンのロールアウト

1. Packer テンプレートを更新します（例：新しい Nginx バージョンのインストール、更新された静的ファイルのコピー）。
2. `packer build` を実行して **新しい** AMI（`ami-0new123ghi789`）を生成します。
3. `main.tf` 内の `ami` フィールドを `ami-0new123ghi789` に変更します。
4. `terraform apply` を実行します。Terraform は古いインスタンスを破棄し、新しいイメージから新しいインスタンスを作成します。

**インスタンスがその場でパッチ適用されることは決してありません。** すべての変更は完全な置き換えです。

### ステップ 4: ブルー/グリーンデプロイ（本番パターン）

ダウンタイムなしで更新するには、Terraform で 2 つの別々の Auto Scaling Group（ASG）または起動テンプレートを定義します。

- **ブルー** = 現在のバージョン（v1）
- **グリーン** = 新しいバージョン（v2）

グリーン ASG をデプロイした後、ヘルスチェックを実行し、Application Load Balancer（ALB）のターゲットグループをブルーからグリーンに切り替えます。トラフィックが安定したら、ブルー ASG を終了します。

## 課題とアンチパターン

- **可変状態：** データベースやその他のステートフルシステムは、完全にイミュータブルとして扱うことはできません。状態はコンピュートレイヤーの外部（例：RDS、スナップショット付き EBS ボリューム、Persistent Volume Claim を使用した Kubernetes StatefulSet）に分離する必要があります。
- **起動時間：** 完全な OS イメージをビルドするのは、ホットパッチを適用するよりも時間がかかります。コンテナを使用するとこの時間は大幅に短縮されますが、大きな VM イメージは依然として扱いにくい場合があります。
- **イメージサイズ：** 規律を守らないと（マルチステージ Docker ビルド、クリーンアップスクリプト）、イメージが肥大化し、デプロイが遅くなります。
- **デバッグ：** SSH アクセスがないため、デバッグは構造化ログ（ELK、CloudWatch、Loki）と分散トレーシング（OpenTelemetry）に完全に依存します。
- **シークレット管理：** シークレットをイメージに焼き付けてはいけません。起動時に Vault、AWS Secrets Manager、CSI ドライバーを介して注入する必要があります。

## 結論

Immutable Infrastructure は、運用の複雑さを本番環境で事後的に管理するのではなく、ビルドパイプラインへと **左側** にシフトさせます。CI/CD やツール（Packer、Terraform、Kubernetes）への初期投資は必要ですが、構成ドリフトや環境の不一致によって引き起こされる障害のクラス全体を排除します。これは、最新のクラウドネイティブ運用の基盤であり、信頼性が高く、安全で、スケーラブルなマイクロサービスアーキテクチャの前提条件です。
