# CAP定理（布鲁尔定理）

## 什么是CAP定理？

CAP定理是分布式系统设计中的一项基础原则。它最早由**埃里克·布鲁尔**在**2000年**的ACM分布式计算原理研讨会（PODC）上提出，并由**塞思·吉尔伯特**和**南希·林奇**在**2002年**正式证明。

该定理指出，一个分布式数据存储在任意时刻只能提供**三个保证中的两个**：
- **一致性（C）**
- **可用性（A）**
- **分区容忍性（P）**

虽然常被过度简化为严格的“三选二”选择，但正确的解释是：**在网络分区存在时，你必须在一致性和可用性之间做出选择**。由于分布式系统中网络分区不可避免，因此你无法同时拥有全部三个属性。

---

## 三个属性

### 一致性（C）
每次读取都能获得**最新的写入**或一个错误。系统中所有节点在相同的逻辑时间看到相同的数据。这要求操作具有全序关系（线性一致性）。

- **影响：** 更强的一致性通常需要在确认写入前在节点间进行同步。
- **示例：** 从任意节点读取的结果必须与从主节点读取的结果相同。

### 可用性（A）
系统中每个非故障节点收到的请求**都必须得到响应**。响应可能不包含最新数据，但不能是错误（如超时或503）。

- **影响：** 即使某些副本不同步，系统仍然保持运行并接受流量。
- **示例：** 即使下游数据库节点不可达，Web应用程序仍能继续提供产品目录。

### 分区容忍性（P）
尽管网络节点间**任意数量的消息被丢弃或延迟**（包括网络分裂、光缆断开、丢包），系统仍能继续运行。

- **影响：** 即使节点间无法通信，系统也必须正确运作。
- **现实：** 在任何地理分布的系统中，分区都是不可避免的。因此，**每个分布式系统都必须具备P容忍性**。

---

## 真正的权衡：CP与AP

因为网络分区（P）在分布式系统中无法避免，所以获取**CA**（一致性 + 可用性）而不考虑分区容忍性在分布式环境下是不可能的。实际的选择是：

### CP系统（一致性 + 分区容忍性）
- **牺牲：** 分区期间的可用性。
- **行为：** 无法保证与集群其余部分一致的节点拒绝响应请求（变得不可用），直到分区解决。
- **应用场景：** 银行账本、库存管理、健康记录——不允许出现陈旧数据的情况。
- **著名例子：**
  - **Apache ZooKeeper**（领导者选举，配置数据）
  - **Apache HBase**（强一致性模型）
  - **MongoDB**（使用 `w: "majority"` 写入关注并从主节点读取）
  - **Redis**（严格一致性保证的集群模式）

### AP系统（可用性 + 分区容忍性）
- **牺牲：** 分区期间的一致性。
- **行为：** 所有节点仍然可用并可独立接受写入。系统依赖冲突解决机制（如最后写入胜利、CRDT）在分区恢复后协调数据。
- **应用场景：** 社交媒体信息流、内容分发、物联网传感器数据、产品目录——对正常运行时间要求极为苛刻的环境。
- **著名例子：**
  - **Apache Cassandra**（可调一致性，默认最终一致性）
  - **Amazon DynamoDB**（多区域最终一致读取）
  - **CouchDB / Couchbase**（多主复制）
  - **Riak**

### CA系统（一致性 + 可用性）
- **上下文：** 仅在非分布式（单节点）系统中可行，或完全忽略分区的系统（这很危险）。
- **著名例子：**
  - 独立运行的 **MySQL** 或 **PostgreSQL** 实例。
  - 在单台服务器上运行的传统ACID关系数据库。
  - *注意：* 在分布式部署中，这些系统必须复制数据，因此不可避免地会遇到分区，迫使它们进入CP或AP行为。

## 关键特征与细微之处

### 1. “P”不是可选的
初学者常犯的错误是设计一个“CA”分布式系统。一旦数据通过网络进行复制，你就容易受到分区的影响。任何真实的分布式系统**都必须**容忍分区，使得在分区发生时真正的选择是**CP与AP**。

### 2. 可调性
现代数据库并不固定在单一分类上。你通常可以在不同的查询中权衡一致性与可用性（或反之）。

- **Cassandra：** 在每个请求中切换 `QUORUM`（强一致性）和 `ONE`（最终一致性）。
- **MongoDB：** 配置 `writeConcern` 和 `readPreference` 在强一致性和弱一致性之间转换。
- **DynamoDB：** 在读取时将 `ConsistentRead` 设为 `true` 或 `false`。

### 3. “三选二”谬误
CAP定理并不是说“系统必须在任何时候都选择两个”。它说的是**在网络分区期间**，你必须在**C**或**A**之间做出选择。在其余时间（网络正常时），系统可以同时追求强一致性和高可用性。

这便是 **PACELC定理** 的切入点。

## PACELC扩展（现代观点）

由**丹尼尔·阿巴迪**提出的PACELC扩展了CAP，明确考虑了系统在**健康**状态下（无分区）的权衡。

**PACELC代表：**
- 如果发生**分区** → 在**可用性**和**一致性**之间权衡。
- **否则**（网络健康时） → 在**延迟**和**一致性**之间权衡。

### 为什么PACELC重要
- **健康状态的权衡：** 即使没有分区，你也可以选择等待副本达成一致（高延迟，强一致性），或者快速响应但可能返回陈旧数据（低延迟，最终一致性）。
- **实际配置：**
  - **CP系统（分区期间）：** 牺牲可用性。
    - **E**（否则）：也可能为了一致性而牺牲延迟（例如同步复制）。
  - **AP系统（分区期间）：** 牺牲一致性。
    - **E**（否则）：可能为了低延迟而牺牲一致性（例如异步复制，只读副本）。

## 实际应用与配置

你不能“安装”CAP定理，但你可以配置分布式数据存储来管理其权衡。

### 概念决策逻辑（伪代码）

```python
# 处理请求时检测分区的顶层逻辑

import config

def handle_write_during_partition(data):
    partition_detected = check_network_health()
    
    if partition_detected:
        if config.CAP_MODE == "CP":
            # 拒绝写入以维护一致性
            raise ServiceUnavailable("分区期间无法保证一致性。")
        elif config.CAP_MODE == "AP":
            # 本地接受写入；稍后解决冲突
            store_with_timestamp(data, node_id=config.NODE_ID)
            return {"status": "accepted", "note": "最终一致性生效中。"}
    else:
        # 网络健康 -> 标准操作
        return normal_write_operation(data)
```

### MongoDB：按查询调整CP/AP

```javascript
// CP行为：确保写入被多数节点确认后才应答
db.inventory.insertOne(
   { item: "journal", qty: 25, status: "A" },
   { writeConcern: { w: "majority", wtimeout: 5000 } }
);

// CP行为：从主节点读取（最强一致性）
db.inventory.find({ status: "A" }).readPref("primary");

// AP行为：从任意副本节点读取（可能得到陈旧数据）
db.inventory.find({ status: "A" }).readPref("secondary");

// AP行为：如果主节点不可达，则允许从副本节点读取
db.inventory.find({ status: "A" }).readPref("secondaryPreferred");
```

### Apache Cassandra：可调一致性级别

```cql
-- 强一致性（倾向于CP）
-- 确保仲裁中的所有副本拥有相同数据
SELECT * FROM users WHERE user_id = 123 CONSISTENCY QUORUM;

-- 强一致性写入
INSERT INTO users (user_id, name) VALUES (123, 'Alice') USING TIMESTAMP 1000;
-- 确保仲裁确认了写入
-- 需要一致性级别 QUORUM 或 ALL

-- 最终一致性（倾向于AP，低延迟）
SELECT * FROM users WHERE user_id = 123 CONSISTENCY ONE;

-- 高可用性，低一致性（AP）
-- 仅由一个节点确认写入
INSERT INTO users (user_id, name) VALUES (456, 'Bob') CONSISTENCY ANY;
```

## 何时选择CP与AP

| 场景 | 推荐方式 | 理由 |
|---|---|---|
| 支付处理 / 账本 | **CP** | 不一致的计数或结余会导致财务损失和法律问题。分区时暂时停机比双花更好。 |
| 健康记录 / 医疗数据 | **CP** | 生命攸关的决策依赖于完整、准确的数据。停机比冲突或陈旧诊断更安全。 |
| 用户会话数据（电子商务） | **AP** | 即使一个数据中心离线，用户也必须能够浏览商品并添加到购物车。库存数据暂时过时是可接受的折中。 |
| 社交媒体信息流 | **AP** | 用户期望网站保持在线。丢失一个赞或延迟一条评论，只要应用保持响应，都是可接受的。 |
| 内容分发 / CDN | **AP** | 提供略微过期的缓存页面远比显示错误页面好得多。 |
| 元数据 / 配置存储（ZooKeeper, etcd） | **CP** | 配置必须在集群中具有权威性和一致性。将集群分割成不一致的视图很危险（脑裂）。 |

## 历史与影响

### 时间线
- **1998年：** 埃里克·布鲁尔首次提出三个属性的概念。
- **2000年：** 布鲁尔在PODC上正式提出这一猜想。
- **2002年：** 麻省理工学院的塞思·吉尔伯特和南希·林奇发表《布鲁尔猜想与一致、可用、分区容忍Web服务的可行性》，正式证明了该定理。
- **2000年代末：** 该定理直接影响了**Amazon DynamoDB**、**Google Bigtable**、**Apache Cassandra**和**MongoDB**的架构。
- **2010年代：** NoSQL运动将CAP定理作为核心设计原则。PACELC被引入，以阐明不仅是分区期间，而是“始终”存在的权衡。
- **2020年代：** 现代分布式SQL数据库（Spanner、CockroachDB、YugabyteDB）试图突破界限，通过激进地降低分区发生的概率和持续时间（例如使用TrueTime/紧密时钟同步）来在大多数时间追求“C与A”兼得。

### 关键洞察
CAP定理的革命性在于它为架构师提供了一种形式化的语言来讨论权衡。在CAP之前，运维人员期望分布式数据库的行为与单体数据库完全一样。该定理迫使业界承认**强一致性是有代价的**，而这个代价通常是在故障期间以可用性为代价付出的。

## 局限性与批评

1.  **虚假的二元性：** 批评者认为“C、A、P”并非二元属性。一致性存在程度之分（强、因果、最终、读己之写），可用性亦如此。
2.  **忽略延迟：** 原始的CAP定理没有明确涉及网络健康时的权衡（这由PACELC解决）。
3.  **CA是陷阱：** 许多工程师寻找“CA”分布式系统。实际上，任何在网络间复制数据的系统都必须具备P容忍性。将系统单纯地标记为“CA”往往是营销说辞而非架构实质。
4.  **现代缓解措施：** 像**Google Spanner**这样的数据库使用原子钟和TrueTime API，在*大多数时候*同时实现强一致性和高可用性，将“三选二”的场景缩减为罕见的边缘情况。

## 参见

- **PACELC定理** — CAP的现代扩展，包含延迟权衡。
- **最终一致性** — 大多数AP系统依赖的一致性模型。
- **ACID vs BASE** — ACID（原子性、一致性、隔离性、持久性）与BASE（基本可用、软状态、最终一致性）。
- **埃里克·布鲁尔** — 定理的原始提出者。
- **分布式系统设计** — 分片、复制、共识算法（Raft、Paxos）。
- **CRDT（无冲突复制数据类型）** — 在AP系统中自然解决冲突的数据结构。
