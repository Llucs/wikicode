# マイクロサービスの堅牢性パターン

マイクロサービスアーキテクチャは、アプリケーションを個別のビジネス機能ごとに独立してデプロイ可能なサービスに分解します。各サービスは明確なAPIを使って他のサービスと通信します。しかし、このアーキテクチャは、サービス間の相互作用に関する新しい課題を引き込み、特に堅牢性と障害耐性に関する課題があります。堅牢性パターンは、マイクロサービスベースのアプリケーションの堅牢性と信頼性を確保するための設計パターンです。

## マイクロサービスの堅牢性パターンの主な特徴

1. **分散制御**: サービスは中央管理されていないため、障害対策が難しい。
2. **非同期通信**: サービス間は非同期メッセージを使って通信するため、遅延や不確実性が生じる。
3. **サービス隔離**: 一つのサービスでの障害は他のサービスの安定性に影響を与えない。
4. **障害耐性**: 部分的な障害がある場合でもシステムは機能を維持しなければならない。

## マイクロサービスの主な堅牢性パターン

### 1. バULKHEAD パターン

- **説明**: バULKHEAD パターンは、一つのサービスが障害を起こした場合に他のサービスへの影響を最小限に抑えるために使用されます。
- **主な特徴**: サービス隔離、コンパッティブレーカー、タイムアウト。
- **実装**: コンパッティブレーカーを使用して故障サービスを隔離し、サービスが回復するまでさらにリクエストを防止します。
- **使用例**: データベース障害、第三者サービスAPI障害、ネットワーク障害。
- **基本的な使用法**: 遠隔サービス呼び出しにタイムアウトを設定し、コンパッティブレーカーを使用してリクエストを防止します。

### 2. コンパッティブレーカー パターン

- **説明**: コンパッティブレーカー パターンは、サービスが第三者サービスから過度に圧迫されるのを保護する戦略です。
- **主な特徴**: 監視、閾値、開く/閉じる状態。
- **実装**: 遠隔サービスの成功率を監視し、成功率が閾値未満になると回路を開きます。
- **使用例**: API障害、データベース障害、ネットワーク障害。
- **基本的な使用法**: 失敗したリクエスト数の閾値を設定し、回路を開き、リクエストを遠隔サービスへ送信を停止します。サービスが回復した後に回路を閉じます。

### 3. フォールバック パターン

- **説明**: フォールバック パターンは、遠隔サービスが障害を起こした場合のデフォルト応答を提供します。
- **主な特徴**: デフォルト応答、キャッシング。
- **実装**: 远隔サービスが障害を起こした場合にキャッシュされたまたは事前定義された応答を返します。
- **使用例**: データベース障害、ネットワーク障害。
- **基本的な使用法**: 遠隔サービスからの応答をキャッシュするか、サービスが利用不可の場合の代替応答を提供します。

### 4. リジェンティック リトライ パターン

- **説明**: リジェンティック リトライ パターンは、失敗したリクエストを遅延後に再試行するためのパターンです。
- **主な特徴**: 指数的な遅延バックオフ、ジョッジ、リトライ。
- **実装**: 失敗したリクエストを遅延後に再試行し、各再試行時に遅延を指数関数的に増加させ、固定のジョッジを追加して「バッファラミングの羊群現象」を避けるために使用します。
- **使用例**: ネットワーク障害、一時的なデータベースロック。
- **基本的な使用法**: 失敗したリクエストを再試行するリトライポリシーを実装し、リトライが失敗した場合に遅延を指数関数的に増加させ、固定のジョッジを追加します。

### 5. ロードシェーディング パターン

- **説明**: ロードシェーディング パターンは、サービスへの負荷を軽減するためにリクエストを落としたり遅らせるパターンです。
- **主な特徴**: タスキング、キューイング。
- **実装**: 入力リクエストを処理するためにキューイングシステムを使用し、サービスが過負荷である場合にリクエストを落としたり遅らせる。
- **使用例**: 高いトラフィック、サービス過負荷。
- **基本的な使用法**: 过負荷時に入力リクエストを管理し、リクエストを落としたり遅らせるためのキューイングシステムを実装します。

### 6. バULKHEAD と コンパッティブレーカーの組み合わせ

- **説明**: バULKHEAD と コンパッティブレーカーを組み合わせることで、マイクロサービスの堅牢なソリューションを提供できます。
- **主な特徴**: サービス隔離、障害耐性。
- **実装**: バULKHEAD を使用してサービスを隔離し、コンパッティブレーカーを使用して一つのサービスでの障害が他のサービスに影響を与えないようにします。
- **使用例**: 複雑なマイクロサービスアーキテクチャ、重要なシステム。
- **基本的な使用法**: バULKHEAD とコンパッティブレーカーを実装し、一つのサービスでの障害が他のサービスの安定性に影響を与えないようにします。

## インストールと基本的な使用法

### インストール

1. **コンパッティブレーカー**:
   - **ライブラリ**: Spring Cloud Netflix Hystrix, Resilience4j, Netflix Ribbon。
   - **例（Spring Cloud Hystrix）**:
     ```java
     @Autowired
     private HystrixCommand.Setter setter;
     
     @HystrixCommand(fallbackMethod = "fallbackMethod")
     public String getResponse() {
         // 远隔サービス呼び出し
     }
     
     public String fallbackMethod() {
         return "Fallback Response";
     }
     ```

2. **バULKHEAD**:
   - **ライブラリ**: Resilience4j, Hystrix。
   - **例（Resilience4j）**:
     ```java
     @Autowired
     private RateLimiter rateLimiter;
     
     @Override
     public String fetchSomeData() {
         return rateLimiter.executeWithRateLimiter(() -> remoteService.getData(), 5);
     }
     ```

### 基本的な使用法

1. **コンパッティブレーカー**:
   - コンパッティブレーカーを遠隔サービスの成功率を監視し、成功率が閾値未満になると回路を開くように構成します。
   - 遠隔サービスが利用不可の場合のデフォルト応答を返す代替メソッドを実装します。

2. **バULKHEAD**:
   - バULKHEAD を遠隔サービス呼び出しを隔離し、並行リクエスト数を制限するように設定します。
   - キューイングシステムを使用して入力リクエストを管理し、サービスが過負荷である場合にリクエストを落としたり遅らせるようにします。

## 結論

堅牢性パターンは、堅牢で信頼性の高いマイクロサービスアプリケーションを構築するための重要なものである。これらのパターンを実装することによって、障害を滑らかに処理し、故障が存在してもシステムが常に可用性を維持できるようにすることができます。パターンの選択は、アプリケーションの特定の要件と関与するサービスの性質によって異なります。
